# Topic: Web Request Lifecycle and Security Monitoring

## 1. Definition
The web request lifecycle is the end-to-end process from domain resolution to HTTP response delivery.  
It includes DNS resolution, network communication, traffic routing, server processing, and response generation.

## 2. Key Concepts
- DNS Resolution (Layer 7 → Layer 3 mapping): Translates domain names to IP addresses  
- Recursive DNS Server: Resolves queries on behalf of clients  
- Authoritative DNS Server: Final source of domain-to-IP mapping  
- HTTP/HTTPS (Layer 7): Application layer communication protocols  
- TCP (Layer 4): Establishes connection using three-way handshake  
- Load Balancer: Distributes traffic across multiple backend servers  
- CDN (Content Delivery Network): Caches and delivers static content from edge nodes  
- X-Forwarded-For: Header containing original client IP behind proxies/CDN  
- Web Server: Software serving HTTP requests (Apache, Nginx, IIS)  
- Virtual Host: Configuration mapping domains to server resources  
- Static Content: Files served directly (HTML, CSS, JS, images)  
- Dynamic Content: Generated by server-side code (PHP, Python, Node.js)  
- Database: Backend data storage (MySQL, PostgreSQL, MSSQL, MongoDB)  
- WAF (Web Application Firewall): Filters HTTP traffic at Layer 7  

## 3. How It Works
1. User enters a domain in the browser  
2. Local cache is checked for existing DNS record  
3. Query sent to recursive DNS server  
4. Recursive server queries root server → TLD server → authoritative DNS  
5. Authoritative DNS returns IP address  
6. Client establishes TCP connection (Layer 4)  
7. HTTPS triggers TLS handshake (encryption setup)  
8. HTTP request is sent to server (Layer 7)  
9. Traffic passes through CDN and/or Load Balancer  
10. Load Balancer forwards request to backend server  
11. Web server processes request:
   - Static content → returned directly  
   - Dynamic content → executed via backend logic + database  
12. Response returned to client via same path  

## 4. Practical Use (Security Context)
- Monitor DNS logs for:
  - High-entropy subdomains → possible data exfiltration  
  - High query volume → possible DGA activity  
- Detect DNS tunneling via:
  - Large DNS responses  
  - Repeated queries to same domain with varying subdomains  
- Validate client IP using:
  - X-Forwarded-For header behind CDN/LB  
- Identify reconnaissance:
  - Requests to IP instead of domain  
  - Default page responses (misconfigured virtual hosts)  
- Monitor HTTP parameters for:
  - SQL Injection patterns (`' OR 1=1--`)  
  - XSS payloads (`<script>`)  
- Analyze WAF logs for:
  - DROP actions  
  - Rate limiting events (429)  
- Detect attacks via HTTP status codes:
  - 403 Forbidden → blocked access attempts  
  - 429 Too Many Requests → brute-force or scanning  

## 5. Example
Request:
GET /index.php?name=admin HTTP/1.1  
Host: example.com  

Malicious Request:
GET /index.php?name='; DROP TABLE users;-- HTTP/1.1  

WAF Log Entry:
TIMESTAMP: 2023-11-20 14:20:10  
SRC_IP: 192.168.1.50  
ACTION: DROP  
HTTP_STATUS: 429  
REASON: RATE_LIMIT_EXCEEDED  
REQUEST: POST /login.php  

CDN Scenario:
Source IP in logs: 10.x.x.x (Load Balancer)  
Original IP: Present in X-Forwarded-For header  

## 6. Key Takeaways
- DNS is the first observable stage of potential attack activity  
- Load Balancers and CDNs obscure true client IP  
- X-Forwarded-For is critical for attribution  
- Virtual host misconfiguration exposes reconnaissance attempts  
- Dynamic content introduces injection attack surface  
- WAF provides Layer 7 filtering and attack visibility  
- Correlating DNS, HTTP, and WAF logs is essential for detection
